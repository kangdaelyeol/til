## 운영체제의 개요

### 운영체제의 정의

운영체제는 컴퓨터 사용자와 컴퓨터 하드웨어 간의 Interface로써 동작하는 시스템 SW다.

OS는 **스스로 기능을 수행하지 않고** , 다른 응용 프로그램이 작업을 할 수 있도록 환경을 마련한다.

### OS 성능 평가 항목

**`처리량(Throughput)`**

주어진 시간 내에 처리되는 작업의 양.

Throughput이 높을수록 OS 성능이 좋다.

**`응답 시간(Response Time)`**

사용자가 시스템에 명령을 입력한 뒤 그 결과가 출력될 때까지 걸리는 시간.

Response Time이 짧을수록 OS 성능이 좋다.

**`신뢰성(Reliability)`**

주어진 작업에 대해서 얼마나 오류 없이 처리하는 지에 대한 척도.

**`사용 가능도(Availability)`**

시스템 운영 시간 중 얼마나 많은 시간을 사용 가능한지에 대한 척도.

### OS 기능

메모리 상태관리, 사용자간의 자원공유 등의 기능을 제공.

### OS 운영 방식

**`일괄처리 시스템(Batch Processing System)`**

유사한 성격의 작업을 한 번에 모아 처리하는 시스템.

오프라인 시스템에서 사용

**`다중 프로그래밍 시스템(Multi Programming System)`**

CPU처리 효율을 극대화 하기 위해 하나 이상의 프로그램을 RAM에 적재하고 처리하는 방식

**`시분할 시스템(Time Sharing System)`**

**하나의 컴퓨터를 여러개의 단말기** 가 공동으로 사용하는 시스템.

단말기 사용자를 위한 대화형 처리를 위해 개발되었으며, **실시간 응답** 이 요구된다.

**`다중 처리 시스템(Multi Processing System)`**

**여러 개의 CPU** 와 한 개의 주기억장치로 여러 프로그램을 동시에 처리하는 시스템.

**`실시간 처리 시스템(Real Time Processing System)`**

작업이 발생한 시점에서 즉각적으로 처리하는 시스템.

**`다중 모드 시스템(Multi Mode System)`**

일괄처리, 시분할, 다중처리, 실시간 처리를 함께 사용하는 시스템

**`분산 처리 시스템(Distributed Processing System)`**

여러 대의 컴퓨터들에 의해 작업을 나누어 처리하고, 결과를 통신망을 이용해 공유하는 시스템.

### OS 구성

OS는 제어 프로그램(Control Program)과 처리 프로그램(Processing Program)으로 구분할 수 있다.

제어 프로그램은 감시, 작업제어, 데이터 관리 프로그램으로 구성된다.

처리 프로그램은 언어 번역, 서비스, 문제 처리 프로그램으로 구성된다.

### 제어 프로그램(Control Program)

**`감시 프로그램(Supervisor Program)`**

자원의 할당 및 시스템 전체의 작동상태를 감시, 감독하는 프로그램.

- 인터럽트 처리기 — 인터럽트 발생 시 원인 파악 후 적절한 처리 루틴으로 분기

- CPU 스케줄러 — 어떤 프로세스에 CPU를 줄지 감시·결정

- 메모리 감시 모듈 — 메모리 사용 현황을 감시하고 오류 감지

- 시스템 콜 핸들러 — 사용자 프로그램의 OS 요청을 감시하고 처리

**`작업 제어 프로그램(Job Control Program)`**

스케줄링 및 시스템 자원 할당 기능을 수행.

- 프로세스 스케줄러 — 준비 큐의 프로세스에 CPU 할당 순서 결정 (FCFS, Round Robin 등)

- JCL(Job Control Language) 처리기 — 대형 메인프레임에서 작업 순서와 자원 요구를 기술한 명령
  처리
- 배치 작업 관리자 — 여러 배치 작업의 실행 순서와 우선순위 관리

- 자원 할당 모듈 — 프린터, 디스크 등 자원을 각 작업에 배분

**`데이터 관리 프로그램(Data Management Program)`**

주기억장치와 보조기억장치 간의 자료 전송, 파일의 조작 및 처리, 프로그램 간의 논리적 연결 등 시스템에서 취급하는 파일과 데이터를 표준적인 방법으로 처리할 수 있도록 관리.

- 파일 시스템(FAT, NTFS, ext4) — 파일 생성·삭제·읽기·쓰기를 표준 방법으로 관리

- 버퍼 관리자 — 주기억장치↔보조기억장치 간 데이터 전송 시 버퍼를 통해 속도 차이 완충

- 입출력 관리자(I/O Manager) — 디스크, 테이프 등 장치와의 데이터 입출력 처리

- DBMS(데이터베이스 관리 시스템) — 데이터를 구조화하여 저장·조회·수정 관리

### 처리 프로그램(Processing Program)

**`언어 번역 프로그램(Language Translator Program)`**

원시 프로그램을 번역하여 목적 프로그램을 생성.

Assembler, Compiler가 이에 해당.

**`서비스 프로그램(Service Program)`**

시스템 제공자가 사용 빈도가 높은 프로그램을 미리 작성하여 사용자에게 제공.

시스템 제공자가 미리 만들어 제공하는 유틸리티성 프로그램.

- 정렬/병합 프로그램 — 데이터를 오름차순/내림차순으로 정렬

- 링커(Linker) — 목적 파일들을 연결해 실행 파일 생성

- 로더(Loader) — 프로그램을 메모리에 올려 실행 준비

- 디버거(Debugger) — 프로그램 오류 탐지 및 수정 지원

- 텍스트 에디터 — 코드/문서 편집 (예: vi, notepad)

- 파일 압축 프로그램 — 파일 압축/해제 유틸리티

**`문제 프로그램(Problem Program)`**

특정 업무 처리를 위해 사용자 또는 기업이 직접 작성한 프로그램.

- 급여 관리 프로그램 — 직원 급여 계산 및 명세서 출력

- 재고 관리 프로그램 — 물품 입출고 및 재고 현황 관리

- 성적 처리 프로그램 — 학생 성적 입력, 평균 계산, 출력

- 회계 프로그램 — 기업 매출/지출 내역 관리

- 예약 시스템 — 항공권, 호텔 등 예약 처리

## Unix

### Unix 특징

시분할 시스템을 위해 설계된 대화식 OS.

Multi user, Multi tasking을 지원한다.

90% 이상이 C언어로 구성되어서 이식성(Portability)이 높다.

### Unix 시스템 구성

**`Kernel`**

프로세스 관리, 메모리 관리, I/O 등 핵심적인 역할 수행.

**`Shell`**

사용자로부터 명령을 받아 커널로 전달하는 명령어 해석기.

시스템과 사용자간의 인터페이스.

**`Utility`**

사용자 편의를 위한 프로그램.

vi편집기, 컴파일러, 인터프리터 등이 이에 해당한다.

### Unix File System 구조

**`부트 블록(Boot Block)`**

부팅에 필요한 코드를 저장하고 있는 블록.

**`슈퍼 블록(Super Block)`**

전체 파일 시스템에 대한 정보를 저장하고 있는 블록.

- 전체 블록 수 / 사용 중인 블록 수

- I-node 수, 블록 크기

**`I-node 블록(Index Node Block)`**

각 파일에 대한 정보를 저장하고 있는 블록.

- 파일 타입, 접근 권한(Permission)

- 파일 소유자, 크기, 최종 수정 시간

- 하드 링크 수, 데이터 블록 포인터

I-node에는 **파일명이 포함되지 않음**.

- 파일명은 **디렉토리 블록** 에 저장된다.

**`데이터 블록(Data Block)`**

실제 데이터를 포함하는 블록

### Unix 명령어

**`fsck`**

파일 시스템의 오류를 검사하고 수복(repair)한다.

### Unix 시스템 콜(System Call)

사용자 프로그램이 커널 기능을 요청하기 위해 호출하는 인터페이스.

**`fork`**

새로운 프로세스를 생성한다. 호출한 프로세스의 복사본(자식 프로세스)을 만든다.

**`getpid`**

현재 프로세스의 ID(PID)를 반환한다.

### 환경 변수(Environment Variable)

Shell이 프로그램들 사이에서 값을 전달해주는 역할을 하는 변수.

### 환경 변수 관련 명령어

**`env`**

**전역** 환경 변수를 설정하거나 출력

**`set`**

현재 셸의 **지역 변수(셸 변수)** 와 함수 목록을 출력하거나 셸 옵션을 설정한다.

**`export`**

지역 변수를 **환경 변수(전역)** 로 승격시켜 자식 프로세스에도 전달되도록 한다.

**`unset`**

환경 변수를 제거.

## 프로세스 관리

### 프로세스 제어 블록(PCB - Process Control Block)

OS가 프로세스를 관리하기 위해 프로세스에 대한 정보를 저장해 놓은 블록

프로세스가 생성되면 고유의 PCB가 생성되며, 프로세스가 소멸되면 PCB도 같이 소멸한다.

**`PCB 포함 정보`**

| 항목                 | 설명                                     |
| -------------------- | ---------------------------------------- |
| 프로세스 상태        | 현재 상태 (Ready, Running, Blocked 등)   |
| 프로세스 ID (PID)    | 프로세스를 식별하는 고유 번호            |
| 프로그램 카운터 (PC) | 다음에 실행할 명령어의 주소              |
| 레지스터 정보        | CPU 레지스터 값 (문맥 교환 시 저장/복원) |
| 우선순위             | 스케줄링 우선순위                        |
| 할당된 자원 정보     | 열린 파일, 메모리, 입출력 장치 등        |

### 프로세스 상태 전이

```text
생성(New) --[Admit]--> 준비(Ready) --[Dispatch]--> 실행(Run) --[Exit]--> 종료

실행(Run) --[Time Run Out]--> 준비(Ready)

실행(Run) --[Block / I/O 요청]--> 대기(Blocked) --[Wake Up / I/O 완료]--> 준비(Ready)
```

**`준비 상태(Ready State)`**

프로세스가 CPU를 할당받기 위해 준비하고 있는 상태.

**`실행 상태(Running State)`**

프로세스가 준비 상태에서 CPU를 할당받아 실행중인 상태.

**Dispatch**: 스케줄러가 준비 큐에서 프로세스를 선택하여 CPU를 할당하는 과정.

- 어떤 프로세스를 선택하느냐는 스케줄링 알고리즘(FCFS, Round Robin 등)에 따라 다르다.

**Time Run Out**: 실행 상태의 프로세스가 할당 시간이 종료되는 현상.

**`대기 상태(Blocked state)`**

실행 상태의 프로세스가 I/O 요청 등 특정 이벤트의 완료를 기다리며 CPU를 반납하고 대기하는 상태.

**Block**: 실행 상태에서 대기 상태로 전환.

**Wake Up**: 대기 상태에서 준비 상태로 전환

### 쓰레드(Thread)

프로세스 내의 작업 단위.

커널 스레드, 사용자 스레드로 구분할 수 있다.

| 구분          | 설명                                                            |
| ------------- | --------------------------------------------------------------- |
| 커널 스레드   | OS 커널이 직접 생성·관리. 멀티프로세서 환경에서 병렬 실행 가능. |
| 사용자 스레드 | 사용자 영역의 라이브러리가 관리. 커널은 스레드 존재를 모름.     |

하나의 프로세스는 여러 스레드를 가질 수 있으며, 스레드들은 프로세스의 **코드, 데이터, 힙 영역을 공유** 하고 **스택은 각자 독립적** 으로 가진다.

### 병행 프로세스(Concurrent Process) 오류 방지 기법

**병행 프로세스(Concurrent Process)** 는 두 개 이상의 프로세스들이 동시에 실행 상태에 있는 것을 뜻한다.

**`임계 영역(Critical Section)`**

어느 시점에 반드시 하나의 프로세스만 접근할 수 있는 공유 메모리 영역.

**`상호 배제(Mutual Exclusion)`**

하나의 프로세스가 트랜잭션을 수행하는 동안 접근하는 공유변수에 대해 다른 프로세스가 접근하지 못하게 하는 기법.

**`동기화 기법(Synchronization)`**

**Semaphore**

- Dijkstra가 제안한 방법.

- 프로세스 사이의 동기를 유지하고 상호배제의 원리를 보장.

- 정수형 변수 S를 사용하며, `P 연산(wait)`과 `V 연산(signal)`으로 제어.

**Monitor**

- 모니터의 경계에서 상호배제가 자동으로 시행됨.

- 모니터 외부에서 모니터 내부의 데이터로의 접근을 제한하는 기법.

- Semaphore보다 사용이 쉽고 오류 가능성이 낮음.

### 교착 상태(Deadlock) 발생 조건

**교착상태(Deadlock)** 란, 둘 이상의 프로세스가 서로 다른 프로세스가 차지하고 있는 자원을 동시에 요구하게 되어 무한정 대기하게 되어 프로세스가 중단되는 현상을 말한다.

아래 4가지 조건이 모두 성립하는 경우 교착상태가 발생한다.

**`상호 배제(Mutual Exclusion)`**

한 번에 하나의 프로세스만이 특정 자원을 사용하는 경우.

**`점유 및 대기(Hold and Wait)`**

프로세스가 다른 자원을 할당받기를 기다리는 동안 이미 확보한 자원을 계속 보유하고 있는 경우.

**`비선점(Non-preemption)`**

자원을 확보한 프로세스로부터 다른 프로세스가 그 자원을 빼앗을 수 없는 경우.

**`환형 대기(Circular Wait)`**

프로세스들이 원형(Circular)으로 서로 상대방의 자원을 요구하며 무한정 대기하는 경우.

### 교착 상태 해결 방법

**`예방(Prevention)`**

4가지 발생 조건 중 하나를 제거하는 방법.

자원 낭비가 가장 심한 기법.

**`회피(Avoidance)`**

**교착 상태 발생 가능성을 인정** 하고, 교착 상태가 발생하려고 할 때 피해 가는 방법.

**`발견(Detection)`**

주기적으로 교착 상태가 발생했는지 **검사** 하는 방법.

**`회복(Recovery)`**

교착 상태에 빠진 프로세스를 종료하거나, 자원을 **선점(Preemption)** 하여 다른 프로세스에 할당하는 기법.

## 프로세스 스케줄링

프로세스 스케줄링(Process Scheduling)이란, 프로세스의 생성 및 실행에 필요한 시스템 자원을 해당 프로세스에 할당하는 작업이다.

**다중 프로그래밍** OS에서 자원의 성능을 향상시키고 효율적인 프로세서 관리를 위해 작업 순서를 결정하는 것이다.

### 비선점(Non-Preemptive) 스케줄링 기법

프로세스가 CPU를 할당받으면 다른 프로세스가 CPU를 강제로 빼앗을 수 없고, 사용이 끝날 때 까지 기다리는 방식.

## 기억장치 관리

### 배치 전략

프로그램 또는 데이터를 주기억장치의 어디에 위치시킬 것인지 결정하는 전략

| 전략                 | 내용                                         |
| -------------------- | -------------------------------------------- |
| 최초 적합(First-fit) | 적재 가능한 공간 중 가장 첫 번째 공간에 배치 |
| 최적 적합(Best-fit)  | 단편화 공간이 가장 적게 발생하는 공간에 배치 |
| 최악 적합(Worst fit) | 단편화 공간이 가장 많이 발생하는 공간에 배치 |

**`단편화(Fragmentation)`**

단편화는 메모리 파티셔닝(Partitioning)기법을 기반으로 발생한다.

파티셔닝 기법으로 메모리를 나누었을 때, 프로세스가 메모리를 완전히 활용하지 못하는 현상을 단편화 라고 한다.

| 단편화                | 내용                                                                          | 발생 기법                  |
| --------------------- | ----------------------------------------------------------------------------- | -------------------------- |
| 내부(Internal) 단편화 | 프로세스가 메모리 공간보다 작아서 남은 메모리를 활용하지 못하는 현상          | 페이징(Paging)             |
| 외부(External) 단편화 | 나누어진 메모리 공간보다 프로세스의 크기가 커서 메모리에 할당되지 못하는 현상 | 세그먼테이션(Segmentation) |

**단편화 해결 방법**

| 방법             | 내용                                                              |
| ---------------- | ----------------------------------------------------------------- |
| 압축(Compaction) | 흩어진 빈 메모리 공간을 한 곳으로 모아 큰 연속 공간을 만드는 방법 |
| 통합(Coalescing) | 인접한 빈 메모리 파티션들을 하나의 큰 파티션으로 합치는 방법      |

### 가상기억장치 구현 기법

주기억장치의 부족한 용량을 해결하기 위해 보조기억장치를 주기억장치처럼 활용하는 기법을 가상기억장치(Virtual memory)라고 한다.

가상기억장치의 구현 방법은 페이징(Paging)기법과 세그먼테이션(Segmentation)기법이 있다.

| 기법                       | 내용                                                                                                                                                                |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 페이징(Paging)             | 가상기억장치의 프로세스를 동일한 크기로 나누어 페이지(page)로 구분하고<br /> 주기억장치를 동일한 크기로 나누어 페이지 프레임(page frame)으로 구분하여 적재하는 방법 |
| 세그먼테이션(Segmentation) | 가상기억장치의 프로세스를 논리적 단위(모듈, 스택, 서브루틴 등)로 가변적 분할하여 이를 세그먼테이션이라 정의.<br />각 세그먼테이션을 주기억장치로 적재하는 방법.     |

**`페이지 부재(Page fault) / 스레싱(Thrashing)`**

중앙처리장치에서 페이지를 참조할 때 페이지가 주기억장치에 없는 현상.

주기억장치의 메모리 공간(페이지 프레임)이 모두 사용중일 때 페이지를 교체하기 위해 페이지 프레임 교체 전략을 사용한다.

페이지 부재가 지나치게 발생하여, 프로그램 실행시간보다 페이지 교체 시간이 더 커지는 현상을 **스레싱(Thrashing)** 이라고 한다.

**`구역성(Locality) / 워킹셋(Working set`**

프로세스가 실행되는 동안 프로세스의 일부 페이지만 집중적으로 참조되는 경향성을 **구역성(Locality)** 이라고 한다.

자주 참조되는 페이지들의 집합을 **워킹셋(Working set)** 이라고 한다.

워킹셋 모델은 각 프로세스의 워킹셋을 주기억장치에 유지함으로써 스레싱을 방지하는 전략이다.

| 구역성                | 내용                                       |
| --------------------- | ------------------------------------------ |
| 시간(Temporal) 구역성 | 순환, 서브루틴 등 `반복문` 에 대한 구역성  |
| 공간(Spatial) 구역성  | 배열 순환 등, `순차적 접근` 에 대한 구역성 |
