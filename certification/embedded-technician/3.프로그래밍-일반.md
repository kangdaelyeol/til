## 운영체제 성능 평가 기준

**`처리량(Throughput)`**

단위시간당 처리하는 작업의 양.

**`반환 시간(Turn Around Time)`**

작업 요청으로부터 완료까지 걸리는 시간.

**`응답 시간(Response Time)`**

요청 후 **첫 번째 응답** 이 올 때까지의 시간.

**`신뢰도(Reliability)`**

시스템이 오류 없이 정상 동작하는 정도.

**`가용성(Availability)`**

시스템을 사용할 수 있는 시간의 비율.

가용성 = 가동 시간 /(가동 시간 + 비가동 시간)

## 컴파일러(Compiler)

고급 언어로 작성된 소스 코드를 저급 언어(어셈블리어 또는 오브젝트 코드)로 번역.

소스 코드 전체를 한꺼번에 번역한 후 실행하는 방식.

### 처리 과정

어휘 분석 → 구문 분석 → 의미 분석 → 중간 코드 생성 → 코드 최적화 → 목적 코드 생성

### 관련 개념

**`인터프리터(Interpreter)`**

소스 코드를 한 줄씩 번역하면서 바로 실행.

컴파일러와 달리 목적 파일을 생성하지 않고, 실행 속도는 느리지만 즉시 실행. Python, JavaScript 등이 대표적.

**`크로스 컴파일러(Cross Compiler)`**

현재 사용하는 컴퓨터와 다른 플랫폼에서 실행될 코드를 생성하는 컴파일러. 임베디드 개발에서 자주 사용됨.

PC에서 컴파일해서 ARM 보드에서 실행하는 경우가 대표적.

**`프리프로세서(Preprocessor)`**

컴파일 전에 매크로 치환, 헤더 파일 포함 등의 전처리를 수행.

C언어의 #include, #define 처리가 이에 해당.

## 어셈블러(Assembler)

어셈블리어를 기계어(object code)로 변환.

어셈블리 명령어와 기계어가 거의 1:1 대응되기 때문에 컴파일러보다 변환이 단순함.

### 어셈블러 종류

**`단일 패스 어셈블러(One-pass Assembler)`**

소스 코드를 한 번만 읽어서 번역.

속도가 빠르지만 전방 참조 문제(forward reference problem)가 있어 처리에 제한이 있다.

**`2패스 어셈블러(Two-pass Assembler)`**

소스 코드를 두 번 읽어 번역.

1패스에서 심볼 테이블을 만들고, 2패스에서 실제 기계어로 번역.

전방 참조 문제를 해결할 수 있어 가장 일반적으로 사용됨.

**`크로스 어셈블러(Cross Assembler)`**

크로스 컴파일러와 같은 개념으로, 다른 플랫폼용 기계어를 생성.

## 링커(Linker)

여러 개의 오브젝트 파일과 라이브러리를 하나의 실행 파일로 결합.

각 모듈에서 참조하는 외부 함수나 변수의 주소를 연결해주는 역할을 수행

### 링킹 방식

**`정적 링킹(Static Linking)`**

컴파일 시점에 라이브러리 코드를 실행 파일에 직접 포함시키는 방식.

실행 파일 크기가 커지지만 독립적으로 실행 가능.

**`동적 링킹(Dynamic Linking)`**

프로그램 실행 시점에 라이브러리를 연결하는 방식.

실행 파일 크기가 작고 메모리를 절약할 수 있지만, 실행 시 해당 라이브러리가 있어야 함.

Windows의 DLL, Linux의 .so 파일이 대표적.

---

## 로더(Loader)

실행 파일을 메모리에 적재하여 실행 가능한 상태로 만드는 역할을 수행.

### 기본 기능

**`Allocation(할당)`**

프로그램을 올릴 메모리 공간 확보.

**`Linking(링킹)`**

여러 오브젝트 모듈 간의 외부 심볼 참조를 연결. 링커가 이미 수행했더라도 로더 단계에서 최종 주소 기반으로 재연결이 필요할 수 있음.

**`Loading(적재)`**

실제로 메모리에 올림

### 로더 종류

**`절대 로더(Absolute Loader)`**

프로그래머가 지정한 절대 주소에 프로그램을 적재.

Loading만 수행하고, Allocation과 Linking은 프로그래머가 직접 해야 함. 가장 단순하지만 유연성이 떨어짐.

**`재배치 로더(Relocating Loader)`**

프로그램을 메모리의 어느 위치에든 적재할 수 있는 방식.

상대 주소를 사용하여 적재 시점에 실제 주소로 변환.

Allocation과 Loading을 수행.

**`직접 연결 로더(Direct Linking Loader)`**

Allocation, Linking, Loading을 모두 수행하는 가장 일반적인 로더.

여러 모듈을 연결하면서 메모리에 적재할 수 있다.

**`동적 적재 로더(Dynamic Loading Loader)`**

프로그램 전체를 한꺼번에 올리지 않고, 실행 중에 필요한 부분만 메모리에 적재하는 방식을 사용.

필요한 부분만 메모리에 적재하는 방식을 사용하기 때문에 메모리를 효율적으로 사용할 수 있다.

## 페이지 교체 알고리즘

### FIFO(First In First Out)

메모리에 가장 먼저 들어온 페이지를 교체하는 방식. 구현이 가장 단순하다.

페이지가 메모리에 적재된 순서를 **큐(Queue)** 로 관리하여, 가장 오래된 페이지를 내보낸다.

**`단점`**

자주 사용되는 페이지도 오래되었다는 이유로 교체될 수 있다.

**Belady의 모순(Belady's Anomaly)**이 발생할 수 있다.

- Belady의 모순은 메모리 프레임 수를 늘렸는데 오히려 페이지 부재(Page Fault)가 증가하는 현상이다.

### OPT / Optimal(최적 페이지 교체)

앞으로 가장 오랫동안 사용되지 않을 페이지를 교체하는 방식.

미래의 페이지 참조를 미리 알고 있다고 가정하고, 가장 나중에 사용될 페이지를 내보낸다.

페이지 부재율이 가장 낮아서 이론적으로 최적의 성능을 제공함.

미래의 참조를 미리 알 수 없으므로 실제 구현이 불가능하다. 다른 알고리즘의 성능을 비교하기 위한 이론적 기준으로 사용된다.

### LRU(Least Recently Used)

가장 오래전에 사용된 페이지를 교체하는 방식.

최근에 사용되지 않은 페이지는 앞으로도 사용되지 않을 가능성이 높다는 가정에 기반하여 만들어진 방식.

OPT에 가까운 성능을 보여주며, 실제로 구현 가능한 방식이다.

각 페이지의 **마지막 사용 시간을 기록** 하여, 가장 오래전에 참조된 페이지를 내보낸다.

- 매 참조마다 시간을 기록해야 하므로 오버헤드가 크다.

카운터나 스택을 사용하여 구현하며, 하드웨어 지원이 필요할 수 있다.

Belady의 모순이 발생하지 않는다.

### LFU(Least Frequently Used)

사용 빈도가 가장 낮은 페이지를 교체한다.

**각 페이지의 참조 횟수** 를 카운터로 기록하여, 참조 횟수가 가장 적은 페이지를 내보낸다.

- 카운터 관리의 오버헤드가 있다.

자주 사용되는 페이지를 메모리에 유지할 수 있다.

과거에 많이 사용되었지만 현재는 사용되지 않는 페이지가 계속 메모리에 남아있을 수 있다. 이를 **에이징(Aging) 문제**라고 한다.

### MFU(Most Frequently Used)

LFU의 반대로, 사용 빈도가 가장 높은 페이지를 교체하는 방식.

참조 횟수가 적은 페이지는 최근에 적재되어 앞으로 사용될 가능성이 높다는 가정에 기반한 알고리즘.

일반적으로 성능이 좋지 않아 잘 사용되지 않는다.

### NUR(Not Used Recently)

각 페이지마다 **참조 비트(R)와 변형 비트(M) 2개의 비트** 를 사용하여 4가지 등급을 만들어 우선순위를 정하고 최근에 사용되지 않은 페이지를 교체하는 방식.

LRU를 근사적으로 구현한 방식.

주기적으로 **R비트를 0으로 초기화**하여 최근 참조 여부를 갱신한다. 이 초기화가 없으면 한 번이라도 참조된 페이지는 영원히 R=1이 되어 알고리즘이 정상 동작하지 않는다.

**`NUR 4가지 등급(교체 우선순위 높은 순)`**

- R=0, M=0 → 최근 참조 안 됨, 수정 안 됨(가장 먼저 교체)

- R=0, M=1 → 최근 참조 안 됨, 수정 됨

- R=1, M=0 → 최근 참조 됨, 수정 안 됨

- R=1, M=1 → 최근 참조 됨, 수정 됨(가장 나중에 교체)

LRU보다 오버헤드가 적으면서 비슷한 효과를 얻을 수 있다.

### SCR / Second Chance(2차 기회)

FIFO를 개선한 방식으로, 교체 대상 페이지에게 한 번의 기회를 더 주는 방식이다.

FIFO 순서로 교체 대상을 선택하되, 참조 비트가 1이면 0으로 바꾸고 큐의 뒤로 보내며, 참조 비트가 0인 페이지를 만나면 그때 교체한다.

SCR을 원형 큐(시계 방향 포인터)로 구현한 것이 **클럭(Clock) 알고리즘**이며, 사실상 동일한 알고리즘의 구현 방식 차이다.

FIFO의 단점을 보완하면서 구현이 비교적 간단하다.

### **`페이지 교체 알고리즘 비교`**

| 구분      | 교체 기준                    | Belady 모순 | 오버헤드 | 구현 가능 여부          |
| --------- | ---------------------------- | ----------- | -------- | ----------------------- |
| FIFO      | 가장 먼저 적재된 페이지      | 발생        | 낮음     | 가능                    |
| OPT       | 가장 나중에 사용될 페이지    | 없음        | 없음     | **불가능**(이론적 기준) |
| LRU       | 가장 오래전에 사용된 페이지  | 없음        | 높음     | 가능                    |
| LFU       | 사용 빈도가 가장 낮은 페이지 | 없음        | 중간     | 가능                    |
| MFU       | 사용 빈도가 가장 높은 페이지 | 없음        | 중간     | 가능(잘 안 쓰임)        |
| NUR       | 최근 미참조 + 미수정 페이지  | 없음        | 낮음     | 가능(LRU 근사)          |
| SCR/Clock | FIFO + 참조 비트 2차 기회    | 없음        | 낮음     | 가능                    |

## CPU 스케줄링(CPU Scheduling)

CPU 스케줄링은 어떤 프로세스에게 CPU를 할당할지 결정하는 방법으로, 선점형과 비선점형으로 나뉜다.

### 비선점형(Non-preemptive) 스케줄링

실행 중인 프로세스가 스스로 CPU를 반납할 때까지 기다리는 방식.

구현이 간단하지만 응답 시간이 길어질 수 있다.

**`FCFS(First Come First Served)`**

먼저 도착한 프로세스를 먼저 실행하며, 가장 단순한 방식이다. 따라서 구현이 매우 간단하고 공정하다.

도착 순서대로 큐에 넣고 순서대로 처리한다.

실행 시간이 긴 프로세스가 앞에 있으면 뒤의 짧은 프로세스들이 오래 기다려야 하는 **호위 효과(Convoy Effect)** 가 발생한다. 따라서 평균 대기 시간이 길어질 수 있다.

**`SJF(Shortest Job First)`**

실행 시간이 가장 짧은 프로세스를 먼저 실행한다.

준비 큐에 있는 프로세스 중 CPU 버스트 시간이 가장 짧은 것을 선택한다.

**비선점형 방식 중 평균 대기 시간이 가장 짧으며**, 이론적으로 최적의 비선점형 스케줄링 방식이다.

실행 시간이 긴 프로세스가 계속 밀려나는 **기아 현상(Starvation)** 이 발생할 수 있다.

또한 실행 시간을 미리 정확히 예측하기 어렵다. 실제로는 **지수 평균(Exponential Averaging)** 으로 이전 CPU 버스트 시간을 가중 평균하여 다음 실행 시간을 추정하는 방법이 사용된다.

**`HRN(Highest Response Ratio Next)`**

SJF의 기아 현상을 해결하기 위해 대기 시간을 고려한 방식.

우선순위를(대기 시간 + 서비스 시간) / 서비스 시간으로 계산하여, 이 값이 가장 높은 프로세스를 실행한다.

- 대기 시간이 길어질수록 우선순위가 높아지므로 기아 현상(Starvation)이 해결된다.

짧은 작업과 오래 기다린 작업 모두 공정하게 처리되지만, 매번 우선순위를 계산해야 하는 오버헤드가 있다.

**`우선순위 스케줄링(Priority Scheduling, 비선점형)`**

각 프로세스에 부여된 **우선순위** 에 따라 실행하는 방식.

우선순위는 정적으로 고정되거나 동적으로 변할 수 있다.

중요한 작업을 먼저 처리할 수 있지만, 우선순위가 낮은 프로세스에 기아 현상(Starvation)이 발생할 수 있다.

- 이를 해결하기 위해 **에이징(Aging)** 기법을 사용하여 대기 시간이 길어질수록 우선순위를 높여준다.

**`기한부 스케줄링(Deadline Scheduling)`**

각 프로세스의 **마감 기한(Deadline)** 을 기준으로 스케줄링하여 기한(Deadline) 내에 완료되도록 스케줄링하는 방식.

실시간 시스템에서 시간 제약을 만족시킬 수 있지만, 기한을 지키기 위한 정확한 실행 시간 예측이 필요하고, 스케줄링이 복잡하다.

### 선점형(preemptive) 스케줄링

실행 중인 프로세스를 강제로 중단시키고 다른 프로세스에게 CPU를 넘길 수 있는 방식.

**`Round Robin(RR)`**

FCFS(First Come First Served)에 **타임 슬라이스** 를 적용한 시분할 시스템용 방식.

각 프로세스에 동일한 시간(타임 슬라이스/타임 퀀텀)을 할당하고, 시간이 지나면 큐의 뒤로 보내고 다음 프로세스를 실행한다.

모든 프로세스가 공정하게 CPU를 사용하고 응답 시간이 짧으며, 기아 현상이 발생하지 않는다.

타임 슬라이스가 너무 크면 FCFS와 같아지고, 너무 작으면 문맥 교환(Context Switch) 오버헤드가 커진다. 적절한 타임 슬라이스는 **문맥 교환 시간보다 충분히 크고, 응답 시간 요구사항보다 작아야** 한다.

**`SRT(Shortest Remaining Time)`**

SJF(Shortest Job First)의 선점형 버전으로, 현재 실행 중인 프로세스의 남은 시간보다 더 짧은 프로세스가 도착하면, 현재 프로세스를 중단하고 새 프로세스를 실행한다.

SJF보다 평균 대기 시간이 더 짧지만, SJF와 마찬가지로 기아 현상이 발생할 수 있고, 남은 실행 시간을 계속 추적해야 하는 오버헤드가 있다.

**`선점형 우선순위 스케줄링`**

더 높은 우선순위의 프로세스가 도착하면 현재 프로세스를 중단시키고 더 높은 우선순위를 가진 프로세스를 실행하는 방식.

긴급한 작업을 즉시 처리할 수 있지만, 기아 현상이 발생 가능하다.

**`다단계 큐 스케줄링(Multi-Level Queue)`**

준비 큐를 여러 단계로 분리하여 각 큐마다 다른 스케줄링 방식을 적용하는 방식.

- 예를 들어 전면 작업(대화형)은 Round Robin으로, 후면 작업(배치)은 FCFS로 처리하도록 지정할 수 있다.

- 각 큐 사이에도 우선순위가 있다.

작업 특성에 맞는 스케줄링이 가능하지만, 프로세스가 다른 큐로 이동할 수 없어 유연성이 떨어진다.

**`다단계 피드백 큐 스케줄링(Multi-Level Feedback Queue)`**

다단계 큐(Multi-Level Queue)를 개선하여 **프로세스가 큐 사이를 이동** 할 수 있는 방식.

- 처음에는 높은 우선순위 큐에 들어가고, 타임 슬라이스 내에 완료하지 못하면 낮은 우선순위 큐로 이동한다.

- 반대로 오래 기다린 프로세스는 높은 큐로 올려줄 수 있다.

가장 유연하고 적응적인 스케줄링 방식이다. 짧은 작업은 높은 우선순위 큐에서 타임 슬라이스 내에 완료되어 빨리 끝나고, 긴 작업은 점점 낮은 큐로 내려가 CPU를 덜 받지만 결국 처리되도록 하여 기아 현상을 해결한다.

구현이 가장 복잡하다.

### **`CPU 스케줄링 알고리즘 비교`**

| 알고리즘         | 선점 여부 | 기아 현상 | 특징                                       |
| ---------------- | --------- | --------- | ------------------------------------------ |
| FCFS             | 비선점    | 없음      | 구현 단순, Convoy Effect 발생              |
| SJF              | 비선점    | 발생      | 평균 대기 시간 최소, 실행 시간 예측 어려움 |
| HRN              | 비선점    | 없음      | SJF 개선, 대기 시간 반영 우선순위          |
| 우선순위         | 비선점    | 발생      | Aging으로 기아 해결 가능                   |
| 기한부           | 비선점    | -         | 실시간 시스템용, 정확한 예측 필요          |
| Round Robin      | 선점      | 없음      | 공정, 타임 슬라이스 크기가 성능 결정       |
| SRT              | 선점      | 발생      | SJF 선점형, 평균 대기 시간 가장 짧음       |
| 선점형 우선순위  | 선점      | 발생      | 긴급 작업 즉시 처리                        |
| 다단계 큐        | 선점      | 발생      | 큐 간 이동 불가, 작업 유형별 스케줄링      |
| 다단계 피드백 큐 | 선점      | 없음      | 가장 유연, 구현 복잡                       |

## 프로그래밍 언어

프로그래밍 언어는 하드웨어에 가까운 순서부터 인간에 가까운 순서로 세대(Generation)별로 분류할 수 있다.

### 1세대 언어 — 기계어(Machine Language)

컴퓨터가 직접 이해하는 0과 1의 이진 코드로 된 언어.

CPU가 바로 실행할 수 있어 번역 과정이 필요 없고 실행 속도가 가장 빠르다.

사람이 읽고 작성하기 매우 어렵고, CPU 아키텍처마다 기계어가 다르므로 호환성이 없다.

### 2세대 언어 — 어셈블리어(Assembly Language)

기계어의 이진 코드를 **연상기호(Mnemonic)**로 대체한 저급 언어.

- 연상기호는 ADD(덧셈), SUB(뺄셈), MOV(이동), JMP(점프), CMP(비교), LOAD(적재), STORE(저장) 등이 있다.

기계어와 거의 1:1 대응되므로 하드웨어를 직접 제어할 수 있다.

기계어보다 읽기 쉽지만 여전히 CPU에 종속적이어서 호환성이 낮다.

**어셈블러(Assembler)**에 의해 기계어로 번역된다.

임베디드 시스템, 디바이스 드라이버, 성능이 극도로 중요한 코드에 사용된다.

### 3세대 언어 — 고급 언어(High-Level Language)

인간이 이해하기 쉬운 영어에 가까운 문법을 사용하는 언어.

- 절차지향: C, FORTRAN, COBOL, Pascal
- 객체지향: C++, Java, Python, C#

하드웨어에 독립적이어서 이식성이 높으며, 생산성이 높고 유지보수가 편리하다.

실행을 위해 **번역기(Translator)**를 통해 기계어로 변환해야 한다.

### 4세대 언어(4GL, Fourth Generation Language)

비전문가도 쉽게 사용할 수 있는 매우 높은 수준의 언어.

"무엇을(What)" 할지만 명시하면 "어떻게(How)"는 시스템이 알아서 처리하는 방식을 사용하여 생산성이 매우 높다.

SQL(데이터베이스 질의어), 스프레드시트 매크로, 보고서 생성기 등이 해당한다.

### 5세대 언어(5GL, Fifth Generation Language)

자연어(인간의 언어)에 가깝게 문제를 기술하면 시스템이 스스로 해결 방법을 찾는 언어.

인공지능(AI), 전문가 시스템, 자연어 처리 분야에서 사용된다.

- Prolog, LISP 등이 해당한다.
