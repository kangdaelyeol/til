## 자동 수집 파라미터

### 이벤트 환경 파라미터

**`dl(Document location)`**

| 분류      | 설명                                                                                                                                                            |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 목적      | 이벤트가 실제로 발생한 URL을 기록, meta측에서 이벤트를 특정 URL과 연결하여, 어떤 페이지에서 행동이 발생했는지 추적하고, custom audience의 URL필터링에도 활용됨. |
| 생성 경로 | 픽셀 베이스 코드가 실행될 때 브라우저의 **window.location.href** 값을 자동으로 수집하여 전송                                                                    |

- 픽셀 베이스 코드(Pixel Base Code)는 html **head** 태그 안에 삽입되는 코드다.

**`rl(Referrer Location)`**

| 분류      | 설명                                                                                                                                               |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 목적      | 사용자가 직전에 어디서 왔는지 기록, 유입 경로 분석에 쓰이며, Meta 내부에서 광고 클릭 &rarr; 랜딩 &rarr; 행동으로 이어지는 여정을 추적하는데 활용됨 |
| 생성 경로 | 픽셀 베이스 코드 실행시 자동으로 **document.referrer** 를 읽어서 수집                                                                              |

**`if(iFrame)`**

| 분류      | 설명                                                                                                                                                       |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 목적      | 픽셀이 iFrame 내부에서 실행되었는지 여부 구분 플래그, Meta측에서 데이터 신뢰도 판단에 사용하며, iFrame 내부에서 발생한 이벤트는 일부 기능이 제한될 수 있음 |
| 생성 경로 | 픽셀 베이스 코드 로드시 **window.self === window.top** 조건으로 판별하여 자동 전송                                                                         |

iFrame 내부 픽셀 실행이 신뢰도 측면에서 문제가 될 수 있는 이유는 다음과 같다.

**1. 쿠키 접근 제한**

- iFrame 내부 코드는 메인 페이지의 **\_fbp** 쿠키에 접근하지 못하는 경우가 많다. 즉, 브라우저 보안 정책으로 인해 도메인이 다른 iFrame은 상위 페이지의 쿠키를 읽을 수 없어서, 사용자 식별 정보를 파악할 수 없다.

**2. URL 맥락 왜곡**

- iFrame에서 실행되는 베이스코드는 해당 iFrame의 URL(window.location.href)을 읽게되어 해당 값이 dl값으로 잡힌다. 따라서 사용자 방문 추적에 혼란을 줄 수 있다.

**3. 이벤트 중복**

- 메인 페이지와 iFrame 동시에 픽셀이 있는 경우 이벤트가 중복 집계될 가능성이 있다.

**`it(Pixel Initialize Time)`**

| 분류      | 설명                                                                                 |
| --------- | ------------------------------------------------------------------------------------ |
| 목적      | 픽셀 베이스 코드가 로드 및 실행이 완료된 시각을 기록                                 |
| 생성 경로 | 픽셀 베이스 코드에서 fbq('init', <Pixel ID>) 함수 호출 시 Date.now()를 호출하여 기록 |

**`ts(Timestamp)`**

| 분류      | 설명                                                                                                                                                                                                    |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 목적      | 이벤트가 언제 발생했는지 기록 <br/> 광고 노출, 클릭과 전환 사이의 시간 간격을 계산하여 기여(attribution)를 판단하는 핵심 값<br/>CAPI와 Pixel 이벤트 중복 제거(deduplication)시에도 시간 기준으로 활용됨 |
| 생성 경로 | **Date.now()** 로 밀리초를 얻은 뒤 1000으로 나눠 초 단위 Unix timestamp로 변환하여 전송                                                                                                                 |

ts는 it와 쌍으로 쓰이는데, 두 값의 차이로 픽셀이 준비된 시점부터 실제 이벤트가 발생하기까지 걸린 시간을 측정한다. 단, ts는 초 단위, it는 밀리초 단위이므로 비교 시 단위를 통일해야 한다.

- **(ts × 1000) - it** 값이 비정상적으로 짧으면 사용자 행동이 아닌 봇 또는 스크립트 오류로 인한 자동 발생 가능성을 의심하는데 활용된다.

**`iw(Is Wrapper)`**

| 분류      | 설명                                                                                                                                                                        |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 목적      | 픽셀이 Meta의 공식 래퍼 라이브러리를 통해 실행되었는지 여부를 나타내는 플래그<br/> 서드파티 또는 커스텀 래퍼를 통해 픽셀이 감싸져 실행되는 경우를 구분하기 위한 내부 진단값 |
| 생성 경로 | 픽셀 로드 환경을 자동 감지해서 설정됨, false 값은 래퍼없이 실행되었다는 의미로, 정상 상태를 의미                                                                            |

wrapper 라이브러리로 실행된 코드를 구분하는 이유는 다음과 같다.

**1. 동의 관리(CMP, Consent Management Platform)**

- 메타 픽셀은 쿠키를 기반으로 정보를 수집하므로, 개인정보보호법 등에 의하여 사용자가 동의하지 않으면 픽셀이 실행되면 안된다.

- CMP는 래퍼 역할을 하여 동의한 사용자에게만 픽셀을 실행 하도록 제어할 수 있다.

**2. 커스텀 데이터 가공**

- 픽셀이 실행되기 전에 커스텀 래퍼를 통해 데이터를 가공, 필터링 등 하는 경우가 있다.

- 커스텀 래퍼를 거치는 경우 일부 파라미터가 누락되는 경우가 있어 **iw=true** 인 경우 Meta 내부적으로 신뢰도 가중치를 보정할 수 있다.

- 이벤트에 문제가 생겼을시 픽셀 문제인지, 래퍼 문제인지 구분하기 위한 진단 플래그 역할도 담당한다.

**`tz(Timezone Offset)`**

| 분류      | 설명                                                                    |
| --------- | ----------------------------------------------------------------------- |
| 목적      | 사용자 브라우저의 로컬 시간대를 UTC기준 분(minute) 단위 오프셋으로 기록 |
| 생성 경로 | 픽셀이 new Date().getTimezoneOffset() 메서드를 자동으로 호출하여 수집   |

한국 표준시는 **UTC+9** 이며, **new Date().getTimezoneOffset()** 메서드는 시간이 앞서면 음수를 반환한다. Meta에서는 이를 양수로 변환하여 전송한다.

```js
new Date().getTimezoneOffset(); // -540 (KST, UTC+9 기준)

tz = 540; // 부호 반전해서 전송
```

**`plt(Page Load Time)`**

페이지 접속 후 픽셀코드가 초기화될 때 까지의 시간을 ms로 표현.

브라우저의 Performance API를 사용하여 픽셀이 호출 및 저장.

```js
plt =
	window.performance.timing.domContentLoadedEventEnd -
	window.performance.timing.navigationStart;
```

| 목적 | 이벤트 데이터를 어느 광고 계정 및 픽셀로 귀속시킬지 식별하는 고유 번호 |

| 분류      | 설명                                                                 |
| --------- | -------------------------------------------------------------------- |

### 전환 데이터(Conversion Data)

**cd** 는 Custom Data의 약자로, 이벤트에서 어떠한 일이 구체적으로 발생했는지 Meta에 알려주는 파라미터 묶음이다.

**`cd[contents]`**

상품 상세정보(Contents)를 의미한다.

```js
[
	{
		id: '318.P00000MG000A',
		quantity: 1,
		item_price: '49000.00',
	},
];
```

contents 값이 Meta에 적용하는 방식은 다음과 같다.

**1. 상품 단위 Audience 생성**

- id 값을 기준으로 해당 상품과 상호작용한 사람들의 Audience를 만들 수 있다.

**2. Advantage + catalog campaign 연동**

- Meta상품 카탈로그의 product id와 contents 파라미터의 id값이 일치할 때, 해당 상품을 구매하지 않은 사람에게 광고를 다시 보여주는 **다이나믹 리타겟팅** 이 작동한다.

**3. item_price와 value의 교차 검증**

- 단일 상품의 경우 item_price \* quantity === cd\[value\], 다수 상품의 경우 Σ(item_price × quantity) === cd\[value\] 조건을 만족하는지 Meta가 자체 검증하여 신뢰도를 조정한다.

**`cd[content_type]`**

콘텐츠 유형(Content Type)을 의미한다.

사용할 수 있는 값은 다음과 같다

| 값            | 의미                                                          |
| ------------- | ------------------------------------------------------------- |
| product       | 개별 SKU(Stock Keeping Unit, 특정 색상 및 사이즈의 단일 상품) |
| product_group | 상품 그룹을 의미                                              |

**`cd[value]`**

전환 가치(Conversion Value)를 의미한다.

이벤트 단위가 비즈니스에 기여하는 가치를 Meta에 알려주는 파라미터다.

Meta 알고리즘에 작용하는 방식은 다음과 같다.

**1. 가치 기반 최적화(Value Optimization)**

- 이벤트 발생 횟수가 아닌, value가 높은 이벤트가 발생하도록 최적화한다.

**2. ROAS 기반 입찰 전략**

- ROAS 기반 입찰 전략을 사용하기 위해 value 값이 요구된다. 그렇지 않으면 ROAS계산이 불가능해서 최저비용(Lowest Cost) 입찰만 선택 가능하다.

**`cd[currency]`**

통화 정보(currency)를 의미한다.

이벤트에서 사용된 통화 단위(어느 국가의 화폐 기준인지)를 Meta에 알려주는 파라미터다.

value와 반드시 쌍으로 존재해야 하며, 없으면 광고 계정의 기본 통화로 가정하고 처리한다.

Meta 알고리즘에 작용하는 방식은 다음과 같다.

**1. 글로벌 가치 정규화**

- 여러 통화를 다루거나, 광고주 간 비교 기준을 맞출 때 currency를 기준으로 USD로 환산해서 정규화 한다.

**2. Value Optimization audience 정밀도**

- currency값이 있어야 정확한 금액 기준으로 필터링할 수 있다.

## Specifications

### Attribution(기여)

전환 달성시, 해당 전환이 어떤 광고 덕분에 발생했는지 판단하는 규칙 체계.

**`Attribution Window(기여 윈도우)`**

Meta가 기여를 인정하는 시간 범위.

기여를 인정하는 범위의 기본 값은 다음과 같다.

- **Click-Through** : 광고 클릭 후 7일 이내 전환 &rarr; 해당 광고에 기여 인정

- **View-Through** : 광고 노출 후 1일 이내 전환 &rarr; 해당 광고에 기여 인정 (클릭 없이 노출만으로도 인정)

두 조건이 동시에 충족될 경우, 클릭(Click-Through)이 우선 적용된다.

### 입찰 전략(Bid Strategy)

Meta 광고 경매에서 예산과 전환 목표를 어떻게 운용할지 결정하는 방식.

**1. Lowest Cost (최저 비용)**

- 현재 대부분 기본값으로 사용

- 예산 소진 극대화 + 건당 비용 최소화

- 광고주 개입: 없음 (예산만 설정)

- 전환 데이터 충분히 쌓이기 전, 볼륨 극대화 원할 때 사용

**2. Cost Cap (비용 한도)**

- 전환 1건당 비용 상한선을 직접 지정

- 목표 CPA가 명확할 때 사용

**3. Bid Cap (입찰가 한도)**

- 경매에서 최대 입찰가를 직접 지정

- 가장 수동적인 방식, 전문가용

- 경매 단가를 직접 컨트롤하고 싶을 때 사용

**4. Highest Value (최고 가치)**

- value 파라미터를 필수로 요구(cd[value] + cd[currency] 있을 때 사용)

- 전환 횟수보다 전환 가치(value) 합계 극대화
  - ex) AddToCart 10건(각 10,000원)보다 AddToCart 3건(각 100,000원)을 선호
